# React Frontend Migration Plan

Status: IN_PROGRESS  
Started: 2026-06-18  
Scope: migrate the single Nuxt `/admin` console into a unified React public site, first-install setup wizard, and SaaS admin frontend.

## Current Evidence

- The initial frontend was `web/admin`, a Nuxt 4 SPA with Vue 3, Pinia, `@nuxtjs/i18n`, `@nuxt/icon`, Material Web behind Aoi wrappers, and static output at `web/admin/.output/public`; Phase 7 has now removed that source tree from the current workspace after route parity was established.
- Initial route audit found old Nuxt routes for login, signup, setup, IAM pages, System pages, media pages, server info, versions, and password/invitation flows; those old route files were removed before the wider `web/admin` tree deletion.
- Initial setup routes were `/setup` and `/setup/*`, both backed by `SetupWizard.vue`; those old Nuxt setup entries and the `SetupWizard.vue` component have since been removed after React setup parity was verified.
- Backend setup APIs remain `/api/v1/setup/status`, `/api/v1/setup/schema`, `/api/v1/setup/configs/:step`, `/api/v1/setup/configs/:step/test`, `/api/v1/setup/runs`, `/api/v1/setup/runs/:id/retry`, `/api/v1/setup/runs/:id/logs`, `/api/v1/setup/runs/:id/steps/:step/skip`, and `/api/v1/setup/complete`; React now owns their frontend integration.
- The old Nuxt API surface and locale files have been deleted with the `web/admin` tree. Migrated setup/auth/admin module calls now live in `web/app/app/lib/api`, and current frontend copy lives in `web/app/app/i18n/locales/{zh-CN,en}.json`.
- Backend locale resources remain `configs/locales/{ui,api,validation,system}/{zh-CN,en-US}.yaml`. The current frontend-to-backend locale bridge maps frontend `en` to backend `en-US`.
- Initial Go evidence showed a single static SPA mounted under non-root `webui.mount_path`, default `/admin`, with static assets served directly and non-asset paths falling back to `index.html`.
- Initial Go static mounter and config validation rejected root mount `/`. Phase 6 has since changed the current default to root React SPA hosting while preserving API/probe/plugin route exclusions.
- Initial evidence before the cutover showed Dockerfile, CI, and `scripts/package.py` building `web/admin` and copying `web/admin/.output/public`; Phase 6 has since moved those production paths to `web/app/build/client`.

## Current Phase 4 Setup Slice Evidence

- Go setup center is the authoritative contract for `/setup/*`: React now consumes `/api/v1/setup/status`, `/api/v1/setup/schema`, `/api/v1/setup/configs/:step`, `/api/v1/setup/configs/:step/test`, `/api/v1/setup/runs`, `/api/v1/setup/runs/:id/retry`, `/api/v1/setup/runs/:id/steps/:step/skip`, `/api/v1/setup/runs/:id/logs`, and `/api/v1/setup/complete`.
- React setup route slugs mirror backend `routeSlugForStep`: `config-source`, `diagnostics`, `preflight`, `database`, `cache`, `storage`, `system`, `migrate`, `seed`, `permissions`, `owner`, `site`, `finalize`, and `verify`.
- Backend schema remains the only source for available setup drivers, fields, options, required flags, testability, and skippability. The React setup wizard does not add unsupported database, cache, storage, mail, or system settings.
- First setup step is local language selection. Changing language immediately updates the wizard UI, validation messages, and API `X-Locale` mapping through the shared i18n and API client layers.
- Setup route recovery preserves language-first onboarding: first-time setup-required redirects land on `/setup`, while a non-sensitive setup-language confirmation flag lets later interrupted sessions resume to the backend `currentStep` route.
- Current-step recovery prefers the `routeSlug` returned inside backend status step schema before falling back to the local key-to-slug map, so React does not have to guess backend setup slugs when the status payload already provides them.
- Backend initcenter now orders first-install configuration as storage before database before cache, and optional configuration dependencies are read from each step schema instead of forcing every optional step to depend on database.
- Backend initcenter now separates early system configuration from site display configuration: `system.configure` keeps default locale, IAM issuer, password policy, and default seed behavior before diagnostics; `site.configure` runs after `iam.owner` and owns the exposed product name, version display name, and public base URL fields before optional finalization.
- Backend initcenter no longer exposes `mail.configure` as a first-install step. SMTP and notification settings remain available through the backend System configuration contract instead of occupying the core first-install wizard path.
- React now has a setup-required route gate backed by `GET /api/v1/setup/status`. When the backend reports `required=true`, public, auth, and admin paths redirect to `/setup`; when setup is complete, `/setup` redirects to `/admin` for an active session or `/login` otherwise.
- Setup form state stays in React memory only. Sensitive backend fields are not hydrated from returned values/defaults into browser state, and setup credentials are not written to localStorage or sessionStorage.
- Setup save/test payloads are restricted to backend schema fields with `configPath`. One-time owner and finalization inputs are validated in memory and submitted only through explicit `/setup/runs` fields, not through config-save endpoints or generic run `values`.
- Aoi React now has a reusable `StepWizard` pattern for step progress, status, keyboard focus, responsive mobile overflow, and status labels.
- React setup step progress now uses backend-reported step statuses as the source of truth, keeps unreported backend steps pending instead of marking them succeeded by index, and marks downstream steps blocked only when a dependency is failed or running.
- Setup wizard now exposes retry, skip, run-log reload, completion, submit confirmation, restart-required, step report, run report, and backend password-policy validation states.
- Unit coverage protects setup slug mapping, sensitive field hydration, password policy failures, visible-field payload filtering, and optional finalization fields in the initialization request.
- React setup schema normalization now preserves backend fields declared only inside grouped schema sections, so grouped Go setup steps hydrate defaults and validate required fields without duplicating frontend-only field lists.
- React setup test feedback now renders backend `repairHint`, stale-input warnings when a user changes tested values, and restart-required hints from the setup test result without inventing remediation text.
- React setup save feedback now preserves backend `envManagedPathsOverwritten`, `envManagedPersistence`, `nextAction`, and returned step metadata, and displays env-managed overwrite warnings for the saved paths.
- Playwright coverage verifies first-time setup entry, current-step recovery after setup-language confirmation, setup dependency blocking, test repair hints, stale-test warnings, env-managed save warnings, and exact test/save payloads against backend-supported fields only.
- React setup owner step now adds a client-only password confirmation field for first account creation. The field is validated in memory, rendered through Aoi form errors, and excluded from setup save/run payloads because it is not part of the backend schema.
- Go router-level setup schema integration coverage now fixes the React setup wizard's backend contract for the required first-install path: `storage.configure -> database.configure -> cache.configure -> iam.owner -> site.configure`, route slugs, localized `en-US` labels, grouped backend fields, sensitive owner password handling, and `webui.public_base_url` site configuration.

## Current Phase 4 Auth Slice Evidence

- React public auth routes now include `/login`, `/signup`, `/password/forgot`, `/password/reset`, `/invitations`, and `/invitations/:token`.
- Password recovery uses the backend-exposed `POST /api/v1/auth/password/forgot` contract with only the `email` field. Unknown-email privacy remains backend-owned; the frontend displays the same non-enumerating success state.
- Password reset uses `POST /api/v1/auth/password/reset` with `token` and `newPassword`. The reset form accepts `token` from the query string but does not store tokens or new passwords in browser persistent storage.
- Invitation acceptance uses `POST /api/v1/invitations/:token/accept` with `username`, optional `displayName`, and `password`. It shows success and sends the user back to login instead of inventing a token/session response.
- Debug notification delivery is displayed only when the backend returns `debug: true`. Debug token and URL are rendered on-screen for local workflows but are not written to localStorage or sessionStorage.
- API fallback hardening rejects successful non-JSON API responses so React auth/setup pages cannot treat SPA HTML fallback as valid API data.
- Unit coverage now checks auth schemas for forgot password, password reset, invitation acceptance, and reset-token i18n availability. Playwright route mocks assert the exact frontend payloads sent to password and invitation endpoints.
- Go router-level auth smoke now initializes the app through `/api/v1/setup/runs`, logs in with the React `/login` payload, refreshes tokens, creates invitations through the backend-supported `POST /api/v1/orgs/:orgId/users/invitations` path, accepts invitations through the public token route, runs password forgot/reset with debug delivery, rejects the old password after reset, and logs in with the new password.
- React admin user invitation creation now uses the backend route shape `POST /api/v1/orgs/:orgId/users/invitations`; list and revoke remain on `GET /api/v1/orgs/:orgId/invitations` and `DELETE /api/v1/orgs/:orgId/invitations/:invitationId`.
- React `/login` now responds to backend `api.auth.captchaRequired`, `api.auth.captchaInvalid`, and `api.auth.mfaRequired` error `messageKey` values by rendering localized captcha or MFA inputs. Captcha challenges are loaded through `GET /api/v1/auth/captcha`, and retry payloads submit only the backend fields `captchaId`, `captchaCode`, or `mfaCode` required for the active challenge.
- Playwright coverage verifies the captcha and MFA challenge login retries on desktop and mobile viewports, including exact second-submit payloads and successful transition into `/admin`.
- The final auth parity pass checked Go handler/service request fields against React auth pages for signup, login, password recovery/reset, invitation acceptance, captcha, MFA, and session persistence. Backend service normalization remains the source of truth for trimming and canonicalizing identifiers, while React keeps tokens only in `sessionStorage` and does not persist passwords, reset tokens, captcha codes, or MFA codes beyond form state.
- React login captcha refresh now passes an `AbortSignal` through the shared API client and aborts stale captcha requests on refresh or route unmount, avoiding state updates from obsolete auth challenges.

## Design System Decisions

- Aoi React component layers: tokens -> primitives -> patterns -> templates.
- Tokens use semantic CSS variables and Tailwind v4 theme integration; component code must not hardcode primary colors, spacing, radii, shadows, z-index, or motion timings.
- Radix UI is an accessibility primitive source only. Project code exposes first-party Aoi React components such as buttons, links, forms, dialogs, menus, tables, badges, state blocks, app shells, and markdown prose.
- `lucide-react` is the icon source. Use icon-only buttons only with accessible names.
- Public site uses a soft, bright, modular product language; admin pages use denser, quieter management UI with clear tables, filters, forms, dialogs, empty/loading/error states, and minimal decoration.
- First-install setup uses a clear, trustworthy wizard language with step progress, safe credential handling, connection tests, recovery states, and completion routing.
- Runtime theme settings are limited to a semantic UI token subset: color, contrast, typography scale, spacing, radius, shadow, motion intensity, and light/dark mode. The page must validate WCAG AA contrast, responsive behavior, and reduced-motion safety, and must not allow raw CSS, HTML, script, unsafe URLs, arbitrary Tailwind classes, or unsupported backend persistence.

## i18n Decisions

- React i18n uses i18next and react-i18next.
- Default frontend locale is `zh-CN`; supported frontend locales are `zh-CN` and `en`.
- Backend API locale header remains `X-Locale`, mapping `zh-CN -> zh-CN` and `en -> en-US` until backend locale canonicalization is migrated.
- Setup `system.configure` default-locale fields use backend locale values. React hydrates and syncs `i18n.defaultLocale` through the same mapping so the UI language `en` never persists to the Go config as a backend locale.
- User-facing copy, validation messages, table columns, navigation, loading/empty/error states, SEO metadata, Open Graph copy, accessibility labels, and Markdown blog metadata must live in locale resources.
- Setup wizard copy, step labels, field help, validation messages, connection-test states, recovery copy, progress labels, and completion messages must live in locale resources.
- Local Markdown blog front matter must include `title`, `description`, `date`, `updatedAt`, `slug`, `tags`, `locale`, `draft`, `cover`, and `author`, and must be parsed by `gray-matter` and validated with Zod.
- New pages or components must add matching `zh-CN` and `en` keys. Deleted surfaces must remove obsolete keys in the same phase.
